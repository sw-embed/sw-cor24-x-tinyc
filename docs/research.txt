Thought for 33s
You want an open-source path from C to COR24, starting with the fastest educational prototype, not a production-grade compiler. You also want to know whether your current “compile to another ISA, then translate assembly” trick can be reused for C, especially with inline asm passthrough.

Bottom line
You have four realistic options, in order from quickest demo to most correct long-term toolchain:

Reuse an existing open-source C compiler that emits assembly for some easier source ISA, then translate that assembly to cor24.s.

Modify a small educational C compiler to emit COR24 assembly directly.

Retarget a purposely-retargetable compiler such as lcc.

Build a real GCC/LLVM backend for COR24.

For your stated goal, I would start with #1 or #2, and only move to #3 if you want “real compiler backend” learning without the full pain of GCC/LLVM.

Architecture and decision criteria
The main design question is:

Do you want to build a compiler backend, or do you just want C programs running on COR24 soon?

Those are different goals.

A useful split is:

Goal A: quick demo / educational tool

tolerate poor codegen

tolerate tiny C subset

tolerate odd ABI/calling convention

emit .s, not ELF/object files

Goal B: “real” compiler port

decent register allocation

proper calling convention

debug info, libs, linker story

robust inline asm constraints

For Goal A, the cheapest route is usually:
C frontend → existing assembly target → translator → cor24.s

That is very close to what you already do with Rust.

Option 1 — C to MSP430 (or similar) to COR24 via assembly translation
Why this is the best first experiment
This is the lowest-risk path because you already have the core idea working:

Rust → MSP430 .s

translate MSP430 .s → COR24 .s

You can try to reproduce that for C:

C compiler for MSP430

possibly inline asm or pseudo-ops embedded in the generated assembly

existing translator extended to handle compiler output idioms

GCC supports inline asm, and MSP430 GCC historically supports GNU-style inline assembly syntax. GCC’s general extended asm mechanism is documented officially, and MSPGCC documents MSP430 inline assembly usage specifically. 

Candidate source compilers for this route
GCC targeting MSP430
This is the most obvious match because:

MSP430 is 16-bit and “small embedded C”-ish

you already have a translator pipeline from MSP430-flavored assembly

GCC supports inline asm and target options for MSP430. 

Clang/LLVM targeting MSP430
Less attractive for now because LLVM/Clang still describes MSP430 support as limited / highly experimental. 

SDCC
SDCC is open-source, retargetable, and supports inline assembler blocks, but it does not target MSP430 and is focused mainly on 8-bit MCUs plus a few small architectures. 

So SDCC is more interesting as a retargeting base than as a drop-in “compile to MSP430” solution.

Can inline asm pass through this way?
Yes, in principle, if you choose a compiler that:

emits textual assembly,

preserves inline asm blocks in a recognizable way,

and your translator can detect and reinterpret them.

TinyCC supports .s / .S and GNU-style asm, but its integrated assembler is target-dependent and not something I would count on as a neutral passthrough layer for a new ISA. 

GCC-style inline asm is more mature, but you still have to decide:

Two modes for passthrough
Mode A — literal passthrough comments

Put COR24 directives/opcodes in comments or markers.

Let the translator replace them after C compilation.

Easiest for demo purposes.

Mode B — abuse inline asm

Write target-ISA snippets in MSP430 inline asm syntax or raw strings.

Have translator recognize a fenced convention such as:

asm volatile(
    "; COR24_BEGIN\n"
    "; add3 r1,r2,r3\n"
    "; COR24_END\n"
);
Translator strips the markers and injects real COR24 code.

For a demo, Mode A is safer because you avoid fighting the source compiler’s parser, operand constraints, clobbers, and register syntax.

Pros
Fastest path to “hello world” or freestanding demo

Reuses your existing translator knowledge

Lets you focus on ABI and assembly mapping before compiler internals

Risks
Generated C compiler output may use constructs your translator does not yet handle

Stack frames, spills, comparisons, switch lowering, and calling conventions may be uglier than your Rust-generated subset

Inline asm can become brittle if compiler optimization rearranges things

Recommendation
If you want something working in days rather than months, start here.

Option 3 — Retarget lcc
This is the most academically “correct” answer for your stated educational goals.

lcc is explicitly a retargetable C compiler, and the classic Fraser/Hanson work is famous for exactly this use case. The current public repo still describes lcc as retargetable, and the project docs point to the classic retargeting material. 

Why lcc is attractive
It was designed to be retargeted

It is dramatically smaller in conceptual scope than GCC/LLVM

It teaches backend design, instruction selection, and machine interface structure

Why it may still not be your first move
It is still a “real compiler” port, not just a weekend hack

Documentation and code style are older-school

You will spend time learning lcc’s way rather than just making COR24 code happen

Difficulty estimate
For an experienced systems person, a minimal lcc retarget is plausible as a focused learning project.
It is still substantially more work than “assembly translator after an existing compiler.”

Pros
Best educational backend-retargeting experience

Cleaner than GCC/LLVM

More principled than bolting patterns onto a toy compiler

Risks
Longer ramp-up

More internal machinery than a quick demo needs

Older ecosystem, less tutorial-friendly than modern hobby compilers

My view
If your goal is “learn real retargetable compiler architecture without drowning in GCC/LLVM”, choose lcc.

Option 4 — GCC or LLVM backend
This is the long-term “serious toolchain” answer.

GCC
GCC backend work involves machine descriptions, target headers/macros, calling convention definitions, reload/register allocation interactions, and many target hooks. GCC’s internals docs explicitly describe backend structure and machine descriptions. 

LLVM
LLVM requires target description classes, instruction definitions, register info, lowering, MC layer pieces, and so on. LLVM’s code generator docs are clear that adding a new target means implementing target description classes and understanding LLVM’s codegen pipeline. 

Difficulty
For a hobby/demo compiler target, both are overkill.

Even when successful, the first usable output often takes a while because you need enough backend and ABI support to survive even simple C.

Pros
Most extensible

Easiest eventual path to “real” optimization and tooling

Works well if COR24 grows into a serious ecosystem

Risks
Months, not weekends

Most of your effort goes into framework plumbing instead of COR24-specific learning

Easy to get stuck before first success

My view
Do this only after you already have either:

a working translator-based path, or

a small direct compiler proving your ABI and ISA mapping decisions

Would a C compiler for MSP430 or another target be able to pass through inline assembler this way?
Yes, but with caveats
The good news
GCC supports inline asm generically. 

MSP430 GCC has target-specific inline asm usage. 

TinyCC also supports GNU-style inline asm. 

SDCC supports inline assembler blocks too. 

The bad news
Inline asm is not magic passthrough unless the compiler and assembler accept the syntax.

So for your technique, the safest variants are:

Variant 1: comments as carrier
Generate legal target assembly with comments containing COR24 pseudo-ops.
Your translator reads comments and substitutes COR24 instructions.

Variant 2: fake nop marker sequences
Emit recognizable no-op asm sequences that the translator rewrites into COR24 instructions.

Variant 3: dedicated intrinsic calls
Write C like:

cor24_out(port, value);
Then have the translator replace the call sequence or emitted label with a COR24-specific expansion.

For maintainability, intrinsics or comment-markers beat raw inline asm.

Practical answer
Yes, a C compiler for MSP430 can participate in this style of pipeline, but I would not depend heavily on “transparent inline asm passthrough” for more than tiny demos.

COR24 + chibicc Planning Document
Purpose
This document lays out a practical plan for creating a demo-quality C-to-COR24 compiler path by adapting chibicc to emit cor24.s directly. The goal is educational and incremental: get a small but real subset of C compiling to your FPGA soft CPU without first building a production-grade toolchain.

This plan is designed to answer one central question:

What exactly do I need to provide in order for a chibicc-based backend effort to proceed efficiently?

1. Project Goal and Scope
1.1 Primary goal
Create a modified version of chibicc that can:

parse a useful subset of C

lower it to COR24 assembly text

support enough ABI and runtime behavior to run simple demo programs

produce understandable, inspectable assembly for learning and debugging

1.2 Non-goals for the first version
The first version should not attempt to be:

production-ready

fully optimizing

GCC-compatible

libc-compatible in any serious way

ELF/object-format aware unless you already need that

complete for the C language

1.3 First milestone
The first successful milestone should be:

compile simple freestanding C into cor24.s

assemble and run on the COR24 environment

support simple integer expressions, locals, functions, branches, loops, pointer dereference, and memory-mapped I/O helpers

2. Recommended High-Level Architecture
2.1 Architecture choice
The cleanest educational approach is:

C source -> chibicc frontend + parser + type system -> custom COR24 code generator -> cor24.s

This avoids translating from another ISA and gives full control over the emitted assembly.

2.2 Why chibicc
chibicc is a good fit because:

it is small enough to understand

it already handles parsing, typing, AST construction, and much of the front-end work

its code generation is simple enough to replace or adapt

it is intentionally educational rather than industrial

2.3 Likely internal strategy
There are two practical implementation patterns:

Pattern A: Replace the existing backend directly
fork chibicc

keep most parsing/type logic intact

replace or heavily edit codegen to emit COR24 assembly

Pattern B: Add a new target mode
keep existing codegen for reference

add a new codegen_cor24.c or equivalent

switch backend with a flag or build config

For learning and maintainability, Pattern B is better.

In order to design or implement a COR24 backend, the missing information is not mostly about C syntax. It is mostly about the machine model, ABI, and toolchain boundary.

endianness

stack-related instructions, if any

Best format:

markdown file plus one or two short code examples

PDF/manual excerpt is also useful if available

C. Calling convention / ABI proposal

Need to know or decide:

which registers are callee-saved

At minimum:

5. Backend Design Choices You Will Need to Make
5.1 Register allocation strategy
For the first version, use a simple register discipline rather than trying to build a sophisticated allocator.

Recommended approach:

reserve a few registers for expression evaluation

reserve one register as stack pointer

optionally reserve one as frame pointer

spill aggressively to stack if needed

The first version should favor correctness and clarity over performance.

5.2 Expression evaluation model
Two common choices:

Choice A: stack-machine-like lowering
evaluate subexpressions one at a time

push intermediate values to stack

simple but verbose

Choice B: fixed temporary register discipline
use a small number of temp registers for left/right operands

spill when nesting is deep

better assembly, modest complexity

Recommended: Choice B with frequent spilling.

5.3 Prologue/epilogue format
You need a standard function frame layout.

Typical questions:

save return address manually or use link register?

save callee-saved regs always or lazily?

frame pointer mandatory or optional?

locals addressed off SP or FP?

Recommended initial design:

always create a frame for non-leaf functions

use FP if it makes codegen simpler

treat leaf optimization as future work

5.4 Handling globals and literals
Need conventions for:

string literal placement

read-only versus writable data sections

global symbol naming

relocation model or absolute-address model

For a demo, it is acceptable to use a very simple symbol model if your assembler/linker tolerates it.

8. Questions That Must Be Answered Early
These are the decisions most likely to block progress.

8.1 Is int 24-bit or 32-bit?
This is probably the single biggest language-model question.

24-bit int
Pros:

maps naturally to the machine

simpler arithmetic and storage

educationally elegant

Cons:

less standard in hosted-C expectations

may expose odd edge cases in chibicc assumptions

might require more edits if code assumes 32-bit int

32-bit int
Pros:

more conventional C behavior

may align better with frontend assumptions

Cons:

more expensive runtime and codegen

every arithmetic op becomes multiword or helper-based

For a demo compiler, 24-bit int may be the most honest and practical choice, unless chibicc internals make that too awkward.

8.2 Are pointers 24 bits?
Probably yes, but confirm.

Need to know whether:

all memory is addressable with 24-bit pointers

code and data use the same pointer representation

function pointers need special treatment

8.3 Does COR24 support byte-addressed memory?
If yes, life is easier for C.

If no, then char * semantics and string handling become more complex and need explicit policy decisions.

8.4 Are there compare instructions or only subtract-and-branch patterns?
This shapes how conditional code generation should be written.

8.5 Is multiply/divide native or runtime-assisted?
If absent in hardware, plan helper routines and constrain initial test programs.

10. Risks and Likely Pain Points
10.1 Frontend assumptions versus 24-bit machine reality
Many compiler components are written assuming 8/16/32/64-bit-friendly targets. A 24-bit machine is unusual enough that type sizes, alignment, and constant handling need careful review.

10.2 C semantics around promotions
Even a tiny compiler quickly runs into:

integer promotions

signed versus unsigned compares

shift behavior

pointer arithmetic scaling

You do not need perfect standards compliance for a demo, but you do need consistency.

10.3 Stack frame bugs
Most early compiler failures are ABI/frame/layout bugs, not parser issues.

10.4 Global initialization
If there is no runtime startup code, then even simple globals and string literals require explicit placement and startup behavior decisions.

10.5 Toolchain mismatch
If chibicc emits nice assembly but your assembler/linker expects odd directives or symbol conventions, progress can stall. That is why working assembly samples matter so much.
